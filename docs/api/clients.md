# Clients API

Endpoints to register, list and deactivate Clients. A Client is an end
customer of an Owner's business — a gym member, an academy student, a
clinic patient. Every Client belongs to exactly one Owner: the relationship
is encoded in the URL (`/api/owners/{ownerId}/clients`) for create/list and
in the persisted `owner_id` foreign key.

All endpoints require authentication. The Owner aggregate is the public
entry point; the Client aggregate sits behind the authenticated boundary.

## Domain model

| Field      | Type    | Constraints                  | Notes |
|------------|---------|------------------------------|-------|
| `id`       | UUID    | server-assigned              | generated by `Client.create` |
| `ownerId`  | UUID    | required                     | references the parent Owner; comes from the URL on create |
| `cedula`   | string  | required, non-blank          | tax/national id of the client |
| `name`     | string  | required, non-blank          | personal name |
| `email`    | string  | required, valid email        | uniqueness is **not** enforced (see Caveats) |
| `phone`    | string  | optional                     | free-form |
| `status`   | enum    | `ACTIVE` \| `INACTIVE`       | always `ACTIVE` on creation; flipped by `/deactivate` |

The Client model is intentionally a record with three transition methods —
`create`, `activate`, `deactivate` — that return new instances rather than
mutating. The handler/use case never sees an in-place state change.

## Endpoints

- [`POST /api/owners/{ownerId}/clients`](#post-apiownersowneridclients) — register a client under an owner
- [`GET /api/owners/{ownerId}/clients`](#get-apiownersowneridclients) — list every client of an owner
- [`PATCH /api/clients/{id}/deactivate`](#patch-apiclientsiddeactivate) — soft-deactivate a client

## `POST /api/owners/{ownerId}/clients`

Register a new Client under the given Owner.

**Authentication:** required — `Authorization: Bearer <accessToken>` header.

### Path parameters

| Name      | Type | Description       |
|-----------|------|-------------------|
| `ownerId` | UUID | parent Owner id   |

### Request

```json
{
  "cedula": "1010101010",
  "name": "Juan Cliente",
  "email": "juan@example.com",
  "phone": "+57 311 111 1111"
}
```

| Field    | Type   | Constraints                |
|----------|--------|----------------------------|
| `cedula` | string | required, non-blank        |
| `name`   | string | required, non-blank        |
| `email`  | string | required, valid email      |
| `phone`  | string | optional                   |

### Responses

**`201 Created`**

```json
{
  "id": "11111111-2222-3333-4444-555555555555",
  "ownerId": "8c1b3f2e-7c45-4a12-9a04-9f3e7c8a1234",
  "cedula": "1010101010",
  "name": "Juan Cliente",
  "email": "juan@example.com",
  "phone": "+57 311 111 1111",
  "status": "ACTIVE"
}
```

The Client is created in `ACTIVE` state.

**`400 Bad Request`** — request validation failed (missing field, malformed email).

**`401 Unauthorized`** — access token missing, expired, malformed or revoked.

**`422 Unprocessable Entity`** — domain invariants violated (empty cedula,
empty name, empty email). Mirrors the request validators as a defense layer
on the domain side.

### Example

```bash
curl -X POST http://localhost:8080/api/owners/8c1b3f2e-.../clients \
  -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..." \
  -H "Content-Type: application/json" \
  -d '{"cedula":"1010101010","name":"Juan Cliente","email":"juan@example.com","phone":"+57 311 111 1111"}'
```

## `GET /api/owners/{ownerId}/clients`

List every Client (active and inactive) tied to the given Owner.

**Authentication:** required.

### Path parameters

| Name      | Type | Description     |
|-----------|------|-----------------|
| `ownerId` | UUID | parent Owner id |

### Responses

**`200 OK`** — JSON array; empty if the owner has no clients.

```json
[
  {
    "id": "11111111-...",
    "ownerId": "8c1b3f2e-...",
    "cedula": "1010101010",
    "name": "Juan Cliente",
    "email": "juan@example.com",
    "phone": "+57 311 111 1111",
    "status": "ACTIVE"
  }
]
```

**`401 Unauthorized`** — access token missing, expired or revoked.

### Example

```bash
curl -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..." \
  http://localhost:8080/api/owners/8c1b3f2e-.../clients
```

## `PATCH /api/clients/{id}/deactivate`

Soft-deactivate the Client: status flips from `ACTIVE` to `INACTIVE`. The row
stays in the database — this is not a delete. Calling deactivate again on an
already-inactive client is idempotent (returns the same record).

**Authentication:** required.

### Path parameters

| Name | Type | Description  |
|------|------|--------------|
| `id` | UUID | Client id    |

### Responses

**`200 OK`**

```json
{
  "id": "11111111-...",
  "ownerId": "8c1b3f2e-...",
  "cedula": "1010101010",
  "name": "Juan Cliente",
  "email": "juan@example.com",
  "phone": "+57 311 111 1111",
  "status": "INACTIVE"
}
```

**`401 Unauthorized`** — access token missing, expired or revoked.

**`404 Not Found`** — no Client with that id.

```json
{
  "title": "Cliente no encontrado",
  "status": 404,
  "detail": "No existe un cliente con ID 11111111-..."
}
```

### Example

```bash
curl -X PATCH \
  -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..." \
  http://localhost:8080/api/clients/11111111-.../deactivate
```

## Security and consistency notes

- **Owner-scoped path on create/list.** The `ownerId` in the URL is the
  source of truth for the parent relationship; the handler never reads it
  from the body. This makes the parent unambiguous and easier to reason
  about for authorization rules ("does the caller own this ownerId?")
  when row-level access checks land.
- **Soft delete only.** `/deactivate` flips status; no record is removed.
  This preserves history for audit and reporting.
- **Defense in depth.** Bean Validation enforces request schema (`400`) and
  the `Client` record's compact constructor enforces invariants (`422`).
  Either layer alone is enough to keep an invalid Client out of the
  application; both together protect against accidental gaps.

## Caveats / open items

- **No owner-existence check on create.** `ClientUseCase.register` calls
  `save` directly and trusts the database foreign key. If `ownerId` does
  not exist in `owners`, the FK constraint rejects the INSERT and surfaces
  as a 500 from the handler — not the `404` the OpenAPI summary advertises.
  Tracked for a follow-up: validate owner existence in the use case and
  map to a clean `404` (or `409` if the FK constraint is reframed as a
  conflict).
- **No cedula-uniqueness check.** The OpenAPI summary advertises a `409`
  for "Client cedula already registered for this owner" but
  `ClientUseCase.register` does not enforce it. The use case's own Javadoc
  is explicit: "la cédula puede repetirse entre owners". The 409 is
  aspirational until uniqueness lands at the use-case level.
- **Authorization is authentication-only.** Any holder of a valid access
  token can list or deactivate any client today. Row-level access control
  (caller's owner id matches the path's `ownerId`) is a planned hardening
  step and the reason `ownerId` lives on the URL.
