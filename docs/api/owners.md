# Owners API

Endpoints to register and query the Owner aggregate. An Owner represents a
business (gym, academy, clinic, etc.) that uses Subscript to manage its own
clients and subscriptions. Registration is the public entry point of the
platform — every other authenticated route assumes an Owner already exists.

All endpoints live under the `/api/owners` prefix and accept/return
`application/json`.

## Domain model

| Field             | Type    | Constraints                             | Notes |
|-------------------|---------|-----------------------------------------|-------|
| `id`              | UUID    | server-assigned                         | generated by `Owner.create` at registration |
| `nit`             | string  | required, non-blank                     | tax id; uniqueness is **not** enforced today (see Caveats) |
| `name`            | string  | required, non-blank                     | personal/legal name of the owner |
| `email`           | string  | required, valid email, unique           | login identifier — uniqueness enforced at the use case |
| `phone`           | string  | optional                                | free-form |
| `businessName`    | string  | optional                                | display name of the business |
| `gracePeriodDays` | integer | required, `>= 0`                        | grace window the owner grants to its own clients before suspending their subscriptions |
| `password`        | string  | required, `>= 8 chars` (request only)   | never returned by the API; stored as a BCrypt hash |

The password hash never crosses the API boundary in either direction — the
domain `Owner` keeps the hash, but `OwnerResponse` strips it.

## Endpoints

- [`POST /api/owners`](#post-apiowners) — register a new owner (public)
- [`GET /api/owners/{id}`](#get-apiownersid) — fetch an owner profile (authenticated)

## `POST /api/owners`

Register a new business account.

**Authentication:** none — this is the bootstrap endpoint.

### Request

```json
{
  "nit": "900123456-7",
  "name": "Ana Pérez",
  "email": "ana@gimnasio.com",
  "phone": "+57 300 000 0000",
  "businessName": "Gimnasio Pérez",
  "gracePeriodDays": 5,
  "password": "S3cure!Pass"
}
```

| Field             | Type    | Constraints                  |
|-------------------|---------|------------------------------|
| `nit`             | string  | required, non-blank          |
| `name`            | string  | required, non-blank          |
| `email`           | string  | required, valid email        |
| `phone`           | string  | optional                     |
| `businessName`    | string  | optional                     |
| `gracePeriodDays` | integer | `>= 0`                       |
| `password`        | string  | required, length `>= 8`      |

### Responses

**`201 Created`**

```json
{
  "id": "8c1b3f2e-7c45-4a12-9a04-9f3e7c8a1234",
  "nit": "900123456-7",
  "name": "Ana Pérez",
  "email": "ana@gimnasio.com",
  "phone": "+57 300 000 0000",
  "businessName": "Gimnasio Pérez",
  "gracePeriodDays": 5
}
```

The password is **not** echoed back.

**`400 Bad Request`** — validation failed (missing field, malformed email,
password shorter than 8 characters, negative `gracePeriodDays`).

```json
{
  "title": "Datos inválidos",
  "status": 400,
  "detail": "password: La contraseña debe tener al menos 8 caracteres"
}
```

**`409 Conflict`** — an Owner with the same email already exists.

```json
{
  "title": "Email ya registrado",
  "status": 409,
  "detail": "Ya existe un owner con el email ana@gimnasio.com"
}
```

**`422 Unprocessable Entity`** — domain invariants violated (empty NIT, empty
name, empty email, negative grace period). These are checked again inside the
`Owner` record's compact constructor as a defense-in-depth layer behind
`@NotBlank` / `@Min`.

### Example

```bash
curl -X POST http://localhost:8080/api/owners \
  -H "Content-Type: application/json" \
  -d '{
        "nit": "900123456-7",
        "name": "Ana Pérez",
        "email": "ana@gimnasio.com",
        "phone": "+57 300 000 0000",
        "businessName": "Gimnasio Pérez",
        "gracePeriodDays": 5,
        "password": "S3cure!Pass"
      }'
```

## `GET /api/owners/{id}`

Fetch the profile of a single Owner.

**Authentication:** required — `Authorization: Bearer <accessToken>` header.

### Path parameters

| Name | Type | Description    |
|------|------|----------------|
| `id` | UUID | Owner identifier |

### Responses

**`200 OK`**

```json
{
  "id": "8c1b3f2e-7c45-4a12-9a04-9f3e7c8a1234",
  "nit": "900123456-7",
  "name": "Ana Pérez",
  "email": "ana@gimnasio.com",
  "phone": "+57 300 000 0000",
  "businessName": "Gimnasio Pérez",
  "gracePeriodDays": 5
}
```

**`401 Unauthorized`** — access token missing, expired, malformed, or
revoked (logout / refresh rotation blacklisted it).

**`404 Not Found`** — no Owner with that id.

```json
{
  "title": "Owner no encontrado",
  "status": 404,
  "detail": "No existe un owner con ID 8c1b3f2e-7c45-4a12-9a04-9f3e7c8a1234"
}
```

### Example

```bash
curl -H "Authorization: Bearer eyJhbGciOiJSUzI1NiIs..." \
  http://localhost:8080/api/owners/8c1b3f2e-7c45-4a12-9a04-9f3e7c8a1234
```

## Security notes

- **Public registration.** `POST /api/owners` is intentionally open: it is the
  only way a new business onboards. Anti-abuse (rate limiting, CAPTCHA, email
  verification) is out of scope for v1 and would land as a separate concern
  in front of the handler.
- **Passwords never leave the domain in plaintext.** The handler hashes with
  BCrypt before constructing the `Owner`, and `OwnerResponse` does not expose
  the hash. The plain `password` field exists only on the request schema.
- **Email is the unique key.** `OwnerUseCase.register` performs an
  `findByEmail` check and emits `409` if it already exists. Two registrations
  with the same email cannot succeed.
- **404 vs 401 ordering.** Spring Security validates the access token before
  the handler runs, so an unauthenticated GET always returns `401` and never
  leaks whether the requested id exists.

## Caveats / open items

- **NIT uniqueness is not enforced.** The OpenAPI summary mentions
  "Email or NIT already registered" for the 409, but `OwnerUseCase.register`
  only validates email today. Two owners could share a NIT until a unique
  constraint (DB level + use-case check) is added — tracked for a follow-up.
- **Grace period semantics live on the Owner.** `gracePeriodDays` describes
  what *this owner* allows for its own clients, not anything Subscript itself
  applies. The platform treats it as opaque metadata for now; it will become
  load-bearing when subscriptions ship.
